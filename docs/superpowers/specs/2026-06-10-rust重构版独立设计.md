# Rust 重构版独立设计

**目标**

将当前仓库从「旧脚本版 + Rust / Tauri 重构版并存」的双轨形态，重组为「仅保留 Rust / Tauri 重构版」的单轨仓库。重组完成后，仓库根目录直接承载 Rust 主项目，旧脚本版实现从当前工作树移除，仅保留 Git 历史中的存档价值。

## 背景

当前仓库存在两套并行体系：

- 仓库根目录承载旧脚本版入口、资源和文档。
- `claude-desktop-zh-cn-rs/` 子目录承载新的 Rust / Tauri 重构版。

这会带来几个问题：

- 新用户进入仓库后，默认看到的仍然是旧版结构。
- CI、发布和文档需要同时兼顾两套体系，维护成本高。
- 旧版即将废弃，但当前仓库信息架构仍然以旧版为主。

因此，本次调整的目标不是「继续兼容双轨」，而是让 Rust / Tauri 重构版成为仓库唯一的当前实现。

## 范围

本次重组包含：

- 将 `claude-desktop-zh-cn-rs/` 下的 Rust / Tauri 主项目提升为仓库根目录主结构。
- 删除旧脚本版代码、旧安装入口、旧资源实现和只服务旧版的工程文件。
- 将 GitHub Actions、发布流程、 Pages 入口和开发说明统一切到 Rust / Tauri 版。
- 最终将 README、`docs` 和 Pages 文案改为仅描述 Rust / Tauri 版。

本次重组不包含：

- 改写 Git 历史。
- 保留旧脚本版作为当前工作树中的可运行实现。
- 继续维护旧版安装说明。

## 用户约束

- 分支、提交信息、后续 PR 标题遵守中文 GitHub 规范。
- 严格遵守 `chinese-git-workflow`、`chinese-commit-conventions`、`chinese-documentation`、`chinese-code-review` 这 4 个 skill。
- 分支命名遵循 `type/description` 形式，使用小写英文和连字符。
- 本次功能分支命名使用：`feat/rust-only-repo`
- 提交信息遵循 `type(scope): 中文描述` 形式。
- 文档排版遵循中文技术文档规范，处理中英文、数字、链接与标点的间距。
- 旧版截图资源可以保留，作为新文档和展示素材使用。
- 旧版文档说明不需要先处理，放到迁移后期统一清理。

## 目标状态

重组完成后，仓库应满足以下状态：

### 1. 根目录即 Rust / Tauri 主项目

仓库根目录直接包含：

- `Cargo.toml`
- `Cargo.lock`
- `apps/`
- `crates/`
- `resources/`
- `scripts/`
- Rust / Tauri 对应的 `.github/` 工作流与配置

不再要求用户进入 `claude-desktop-zh-cn-rs/` 子目录才能找到主项目。

### 2. 旧脚本版实现退出当前代码树

以下旧体系将从当前工作树移除：

- 根目录旧安装入口，如旧 `install-mac.command`、旧 `install-windows.bat`
- 旧版 Python 补丁主实现
- 只服务旧版的资源实现
- 只服务旧版的旧构建/发布说明

保留方式仅限 Git 提交历史，不在当前文件树中继续保留可运行旧版。

### 3. 自动化链路全部服务 Rust / Tauri 版

所有工作流改为以新的根目录结构为准：

- Rust 测试从根目录 workspace 执行。
- 前端依赖安装与 Tauri 构建从新的根目录应用路径执行。
- Release 只产出 Rust / Tauri 安装器。
- Pages 只展示 Rust / Tauri 版相关内容。

### 4. 文档单轨化

最终 README、`docs`、Pages 文案只描述 Rust / Tauri 重构版：

- 安装方式
- 开发方式
- 构建方式
- 发布方式

旧版截图资源允许保留，但只作为界面展示素材，不再承载旧版操作说明。

## 方案比较

### 方案 A：根目录整体切换为 Rust 主仓

做法：

- 将 Rust / Tauri 重构版整体提升到仓库根目录
- 删除旧版当前实现
- 统一 CI、发布和文档入口

优点：

- 仓库结构最清晰
- 用户认知成本最低
- 后续维护只围绕一套实现展开

缺点：

- 一次性改动面最大
- 需要系统性修复路径和自动化引用

### 方案 B：保留 `claude-desktop-zh-cn-rs/` 目录，仅删除旧版

优点：

- 迁移风险较低
- 路径修改更少

缺点：

- 主项目长期埋在子目录里
- 仓库结构不符合“Rust 版就是主仓”的定位

### 方案 C：先删除旧版，暂不提升目录

优点：

- 最快完成表面切换

缺点：

- 只是把真正重构推迟
- 后续还要再做一次目录提升和自动化改造

## 选定方案

采用方案 A：根目录整体切换为 Rust 主仓。

原因：

- 最符合「旧版不要了，只要重构版」的目标。
- 能一次性完成结构、认知和维护面的收口。
- 可以避免后续再次搬迁目录和重新改 CI。

## 实施设计

### 阶段 1：建立隔离工作分支

- 从当前主分支切出规范分支：`feat/rust-only-repo`
- 在该分支中完成全部迁移
- 不改写历史，只做当前树重组

### 阶段 2：提升 Rust / Tauri 项目到根目录

将 `claude-desktop-zh-cn-rs/` 中以下内容迁移到仓库根目录：

- Rust workspace 根配置
- `apps/`
- `crates/`
- `resources/`
- `scripts/`
- 相关 `.github/` 工作流与仓库配置
- 该版本自己的 `README` 内容基础

迁移后，原 `claude-desktop-zh-cn-rs/` 目录整体移除。

### 阶段 3：修复所有路径与工程引用

重点修复：

- GitHub Actions 中所有 `working-directory` 和缓存路径
- Rust workspace 相对路径
- Tauri 构建与资源引用
- 脚本对资源和项目根路径的解析
- Pages 对 `docs/` 内容的引用

目标是让整个仓库在新根目录下仍可正常测试和构建。

### 阶段 4：移除旧脚本版实现

在 Rust 主项目已经提升并且路径修复后，移除旧脚本版实现：

- 旧安装脚本入口
- 旧版 Python 补丁实现
- 旧版资源实现
- 只服务旧版的构建辅助文件

此阶段优先删除“影响工程结构和入口”的旧文件，避免继续误导使用者。

### 阶段 5：验证工程可用性

至少完成以下验证：

- `cargo test --workspace`
- Rust workspace 基础编译检查
- 前端构建检查
- Tauri 打包链路基础检查

如果某项验证受当前环境限制无法完整执行，需要在结果中明确说明未验证项及原因。

### 阶段 6：最后统一处理文档与 Pages

在代码和自动化链路稳定后，统一收尾：

- 重写根 `README.md`
- 清理 `docs/` 中旧版说明
- 保留可复用的旧版截图资源
- 调整 Pages 首页内容，只展示 Rust / Tauri 版

这样可以保证迁移过程中，先解决「能不能构建和发布」，再处理「怎么表达给用户看」。

## 风险与控制

### 风险 1：目录提升后大量路径失效

控制方式：

- 先迁移主项目，再统一修路径
- 用搜索和构建验证双重检查路径残留

### 风险 2：CI 与发布流程仍引用旧子目录

控制方式：

- 将 `.github/workflows/` 纳入本次必改范围
- 逐个验证 `working-directory`、缓存路径、产物路径

### 风险 3：文档先删空导致仓库短时间不可理解

控制方式：

- 文档处理放到最后阶段
- 在代码、构建、发布链路稳定后再统一改写

### 风险 4：旧版截图在清理中被误删

控制方式：

- 明确将截图视为可保留资产
- 文档清理时只删旧版说明，不删可复用图片

## 验收标准

满足以下条件即视为完成：

- 仓库根目录直接承载 Rust / Tauri 主项目
- 当前代码树中不再保留旧脚本版实现
- `claude-desktop-zh-cn-rs/` 不再作为主项目子目录存在
- GitHub Actions 与发布流程仅服务 Rust / Tauri 版
- README、`docs`、Pages 最终仅讲 Rust / Tauri 版
- 旧版截图仍可作为展示素材保留

## 非目标说明

本次不追求：

- 保存旧版在当前树中的可运行能力
- 提供旧版兼容安装入口
- 在本次重组中额外引入与「Rust 主仓化」无关的大规模重构

## 实施后的仓库认知

重组完成后，仓库对外应当表达为：

「这是一个基于 Rust / Tauri 的 Claude Desktop 中文安装器项目；旧脚本版已被仓库当前版本正式替代。」
